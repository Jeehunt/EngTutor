# Автоматический деплой при изменениях в Git

## Варианты реализации

### Вариант 1: GitHub Actions (рекомендуется для production)

**Плюсы:**
- Бесплатно для публичных репозиториев
- Интеграция с GitHub
- Логи и история деплоев
- Можно настроить деплой только для определенных веток

**Минусы:**
- Требует настройки GitHub Secrets
- Немного сложнее в настройке

**Как работает:**
1. Push в ветку `main` → триггер GitHub Action
2. GitHub Action подключается к серверу по SSH
3. Выполняет `git pull` и `deploy_server.sh`

### Вариант 2: Git Webhook + простой сервер на сервере

**Плюсы:**
- Полный контроль
- Можно настроить любую логику

**Минусы:**
- Нужно поддерживать сервис на сервере
- Требует настройки webhook на GitHub

**Как работает:**
1. Push в Git → GitHub отправляет POST на ваш сервер
2. Простой HTTP сервер на сервере получает webhook
3. Запускает `deploy_server.sh`

### Вариант 3: Cron job (проверка изменений периодически)

**Плюсы:**
- Простота
- Не требует внешних сервисов

**Минусы:**
- Задержка до деплоя (зависит от интервала cron)
- Лишние проверки, даже если изменений нет

**Как работает:**
1. Cron каждые 5 минут проверяет `git fetch`
2. Если есть изменения → `git pull` и `deploy_server.sh`

## Рекомендация

Для личного проекта рекомендую **Вариант 1 (GitHub Actions)**:
- Надежно
- Бесплатно
- Логи видны в GitHub
- Можно настроить деплой только для ветки `main`

## Текущий подход (ручной деплой)

**Плюсы:**
- Полный контроль
- Можно протестировать перед деплоем
- Нет риска случайного деплоя

**Минусы:**
- Нужно помнить запускать `./scripts/deploy.sh`

## Вывод

**Для начала оставьте ручной деплой** (`./scripts/deploy.sh`). Когда проект стабилизируется, можно добавить автоматический деплой через GitHub Actions.
